# UI Navigation State Machine Concept V1

Обновлено: 2026-05-02  
Статус: концепция архитектуры

## Краткая идея

Навигация пользовательского интерфейса реализуется как формальная машина состояний.

Рендеринг интерфейса опирается на текущее состояние этой машины. Машина состояний реагирует на пользовательский ввод, события прибора, таймауты и системные ошибки. Графическое описание машины является не только документацией, но и исходным артефактом для генерации кода через StateSmith.

## Мотивация

В предыдущем приборе уже использовалась схема переходов между экранами, реализованная при помощи автомата состояний StateSmith и формализованная в виде диаграммы. Этот подход оказался полезным, потому что в процессе работы над уже реализованным в железе устройством появлялись новые функции, режимы и шаблоны поведения.

Диаграмма состояний получилась не совсем простой, но её было намного проще изменять, чем постоянно переделывать большой объём UI-кода. Поэтому в новом проекте этот подход принимается как базовый архитектурный принцип.

## Почему не draw.io

В предыдущем проекте в качестве редактора диаграмм использовался draw.io. Для начального этапа этого хватило, но при усложнении диаграммы проявились ограничения:

- диаграмма плохо подходила для строгой автоматической обработки;
- StateSmith мог ошибаться из-за проблем соединений на диаграмме;
- ошибки соединений возникали слишком легко;
- результаты было неудобно отлаживать;
- изменения плохо просматривались через Git diff;
- ручное редактирование большой диаграммы становилось хрупким.

## Почему PlantUML

Для нового проекта предполагается использовать PlantUML.

Причины выбора:

- диаграмма хранится как текстовый файл;
- изменения хорошо видны в Git diff;
- файл удобно ревьюить и править вручную;
- диаграммы можно генерировать автоматически;
- PlantUML штатно поддерживает state diagrams;
- StateSmith поддерживает PlantUML как один из входных форматов;
- текстовая диаграмма лучше подходит для долгого проекта, где поведение интерфейса будет постепенно уточняться.

## Архитектурное решение

Глобальная навигация UI управляется машиной состояний.

Все переходы между экранами и крупными режимами должны быть представлены на PlantUML-диаграмме. Код отдельных экранов не должен выполнять скрытые глобальные переходы.

Разрешено:

- менять локальный фокус внутри текущего экрана;
- обновлять визуальные элементы;
- показывать локальные подсказки;
- обновлять текст, шкалы, карточки и индикаторы из UI-модели;
- выполнять локальную анимацию, не меняющую глобальное состояние прибора.

Запрещено:

- напрямую переключать экраны из LVGL callback;
- открывать глобальные диалоги в обход state machine;
- менять режим прибора прямо из widget-кода;
- дублировать навигационную логику в отдельных screen-модулях;
- хранить глобальную логику поведения в наборе разрозненных флагов.

## Роль машины состояний

Машина состояний описывает:

- текущий экран;
- вложенные режимы экрана;
- допустимые переходы;
- реакцию на кнопки, энкодер и системные события;
- действия при входе в состояние;
- действия при выходе из состояния;
- временные состояния;
- аварийные состояния;
- поведение при появлении новых функций.

Машина состояний не должна хранить всю предметную модель прибора. Она управляет навигацией и режимами, а данные о сигнале, частоте, уровне, синхронизации и состоянии backend должны приходить из отдельных UI-моделей.

## Поток событий

```text
Кнопка / энкодер / событие прибора
        ↓
ui_event_queue
        ↓
StateSmith UI state machine
        ↓
entry / exit / transition actions
        ↓
ui_presenter
        ↓
LVGL screens / components
```

Обратный поток от LVGL:

```text
LVGL widget event
        ↓
ui_input_adapter
        ↓
абстрактное UI-событие
        ↓
StateSmith state machine
```

LVGL в этой схеме является слоем отображения и локальной интерактивности. Глобальная логика интерфейса находится в StateSmith-машине.

## Верхнеуровневые состояния

Первый уровень машины навигации можно начать с таких состояний:

```text
Boot
StartupSelfTest
HomeScanning
SignalPopup
DetectionsList
SignalDetails
HoldMonitor
VideoPreview
SettingsMenu
ServiceDiagnostics
FatalError
```

Эти состояния покрывают первый вертикальный сценарий прибора:

```text
Startup → Home scanning → Detection → Signal details → Hold monitor
```

## Разделение состояния навигации и данных прибора

Состояния машины не должны кодировать конкретные параметры сигнала.

Плохо:

```text
Signal_433MHz_Strong_New
Signal_145MHz_Weak_Old
Signal_AnalogVideo_Locked
```

Хорошо:

```text
SignalDetails
HoldMonitor
VideoPreview
```

А конкретные данные хранятся отдельно:

```c
typedef struct {
    uint32_t selected_signal_id;
    uint32_t frequency_hz;
    int16_t power_dbm;
    bool video_available;
    bool video_locked;
} ui_context_t;
```

## Рекомендуемый стиль имён

Чтобы диаграммы были стабильными и удобными для генерации кода, вводится единый стиль имён.

Состояния: `PascalCase`

```text
HomeScanning
SignalDetails
HoldMonitor
```

События: `UPPER_SNAKE_CASE`

```text
EV_BTN_BACK
EV_SIGNAL_DETECTED
EV_TIMEOUT
```

Действия: `snake_case()`

```text
show_home_screen()
start_hold_mode()
request_video_preview()
```

Не рекомендуется смешивать стили:

```text
signalDetected
Signal_detected
SIGNAL_DETECTED
EvSignalDetected
```

## Правило для screen-модулей

Screen-модуль отвечает за отображение одного экрана или группы связанных виджетов.

Screen-модуль может:

- создать LVGL-объекты;
- обновить текст, значения и видимость элементов;
- переместить локальный фокус;
- отправить абстрактное событие в UI event queue.

Screen-модуль не может:

- сам перейти на другой глобальный экран;
- сам изменить режим сканирования, удержания или видео;
- сам принять решение о показе системного диалога;
- хранить скрытую глобальную навигационную логику.

## Минимальный первый результат

Первый этап внедрения этой идеи в проект:

- добавить концептуальный документ;
- добавить верхнеуровневую PlantUML-диаграмму;
- зафиксировать стиль имён состояний, событий и действий;
- договориться, что LVGL callbacks не выполняют глобальную навигацию напрямую;
- начать с простого верхнего уровня, не пытаясь сразу описать все вложенные меню.
